# RubiKey 当前产品状态与下一步方向

本文只描述 RubiKey 当前已经实现的前端与功能，以及基于当前实现做出的完成度判断和后续优化建议。

评估基准为 2026-03-20 仓库内 `RubiKey/` 的实际代码与可构建状态。

---

## 1. 项目当前是什么

RubiKey 是一个基于 Electron、React 与 TypeScript 的 Windows 桌面应用。

它的核心目标是：

- 连接兼容的 GAN 智能魔方
- 识别魔方单步转动
- 将转动映射为系统级键盘或鼠标动作
- 以桌面控制台的方式管理连接、映射、运行状态和诊断信息

当前项目已经不是纯概念验证，而是一个具备完整主链路的早期可用版本。

当前主链路为：

`GAN 智能魔方 -> 蓝牙连接 -> move 解析 -> Profile 查找 -> 主进程宏执行 -> Windows 输入`

---

## 2. 当前前端状态

### 2.1 当前界面形态

当前前端已经形成单窗口控制台结构，而不是简单的调试按钮集合。

当前 UI 采用：

- 左侧浮动导航
- 侧边栏底部主题切换入口
- 右侧单工作区内容区
- 首页总览 + 分页查看详细信息

当前一共有 6 个视图：

- 首页
- 方案映射
- 最近转动
- 执行回响
- 连接诊断
- 关于

### 2.2 当前前端已经做到的事情

- 展示设备连接状态
- 展示当前激活方案
- 展示系统运行状态
- 首页提供可选的 GAN MAC 地址输入框，允许在自动获取失败或异常时手动指定
- 支持樱粉和风 / 淡蓝简洁两套界面主题切换，并记住用户选择
- 提供连接、断开、启动/暂停、紧急停止操作
- 管理多套 Profile
- 切换当前激活 Profile
- 新增和删除 Profile
- 为 move 增量添加映射
- 编辑每条规则的动作类型、目标、行为和持续时间
- 展示最近转动记录
- 展示动作执行回响
- 展示连接诊断摘要与详细日志

### 2.3 当前前端的产品特征

从当前页面结构来看，RubiKey 前端已经有比较明确的产品方向：

- 首页负责状态与快速控制
- 首页同时承担 GAN MAC 地址兜底输入
- 方案映射页负责日常配置
- 最近转动页负责输入确认
- 执行回响页负责结果确认
- 连接诊断页负责问题排查

这意味着前端的信息架构已经基本成型，后续重点更像是打磨，而不是重新搭框架。

### 2.4 当前前端还存在的限制

- Profile 编辑仍然偏表单式，不够直观
- 规则卡片缺少更强的动作结果表达
- 首页仍保留方案映射预览，信息还能继续收敛
- 诊断页虽然已有“摘要 + 日志”分层，但仍偏技术视角
- 组件主页面仍命名为 `CubeDebugPage`，与当前产品定位不完全一致

## 3. 当前功能状态

### 3.1 已落地的核心能力

- Electron 主窗口可运行
- 主窗口关闭后会隐藏到托盘
- 托盘菜单支持显示/隐藏、启停系统、紧急停止、退出
- 已注册全局快捷键
  - `Ctrl/Cmd + Shift + F11`：切换运行状态
  - `Ctrl/Cmd + Shift + F12`：紧急停止
- 已支持 GAN 设备连接与断开
- 已支持 GAN `v2 / v3 / v4` 协议分流与通知监听
- 已支持解析 12 个基础单步动作
  - `U / U'`
  - `R / R'`
  - `F / F'`
  - `D / D'`
  - `L / L'`
  - `B / B'`
- 已支持多 Profile 持久化
- 已支持根据当前激活 Profile 执行 move 对应动作
- 已支持动作串行执行
- 已支持紧急停止中断执行并释放已按下输入

### 3.2 当前动作模型边界

当前动作模型仍然保持轻量，便于先把主链路稳定下来。

目前支持：

- 动作类型：键盘 / 鼠标
- 键盘目标：`A-Z`
- 鼠标目标：左键 / 右键
- 行为：`tap` / `hold`
- 长按时长：毫秒级配置

当前还不支持：

- 组合键
- 滚轮
- 连续动作序列
- 更复杂的系统宏

### 3.3 当前配置模型边界

当前配置系统已经够支撑基本使用，但仍然比较简洁。

当前已有：

- 全局启停状态
- 当前激活 Profile
- 多 Profile 列表
- 每个 Profile 的 move 规则

当前没有：

- Profile 说明字段
- 标签、分组、排序等组织能力
- 更高层的场景化模板系统

## 4. 当前完成度总结

### 4.1 总体判断

RubiKey 当前可以定义为：

- 可运行的桌面原型
- 已打通核心业务链路
- 已具备早期使用价值
- 尚未进入稳定交付阶段

### 4.2 完成度拆分

如果按几个维度粗略判断当前完成度：

- 核心主链路完成度：`75% - 85%`
- 前端产品形态完成度：`65% - 75%`
- 工程完备度：`50% - 60%`
- 交付准备度：`45% - 55%`

这个结果表示：

- 功能上已经不是空壳
- 架构上已经有持续迭代基础
- 但稳定性、验证深度和易用性仍有明显提升空间

### 4.3 为什么说主链路已经完成度较高

因为当前代码已经把最关键的几个环节串起来了：

- 能连设备
- 能读协议
- 能解 move
- 能查规则
- 能执行键鼠动作
- 能在 UI 里看到输入、输出和诊断反馈

这条链路一旦成立，后续很多工作都属于增强和打磨，而不是从零突破。

### 4.4 为什么说工程和交付完成度还不高

因为当前仓库内仍然缺少这些关键支撑：

- 自动化测试
- 设备兼容矩阵
- 打包产物验收记录
- 错误恢复策略说明
- 更系统的验证文档

所以当前更适合被描述为“已跑通、可继续打磨”，而不是“已经稳定可交付”。

## 5. 当前工程状态

### 5.1 已确认的工程情况

截至本次整理，仓库内已确认：

- `npm run typecheck` 可通过
- `npm run build` 可通过
- Windows 打包脚本已固定使用仓库内 `.electron-builder-cache/` 与同卷临时目录，减少 `winCodeSign` 缓存下载导致的打包失败
- portable 版已显式将 `userData` 重定向到便携包同目录的 `data/`，避免退出后配置丢失
- 方案编辑已改为自动保存，并在保存失败时直接反馈错误状态，降低手动点击保存遗漏导致的配置丢失风险
- packaged 环境已改为优先使用 CommonJS preload 注入 `window.rubikey`，并在注入缺失时直接给出明确错误提示
- packaged / portable 版的 preload 已改为从 `app.asar.unpacked` 的真实文件路径加载，降低 `window.rubikey` 注入失败概率

这说明：

- TypeScript 类型关系基本闭合
- renderer 与 main 的生产构建链路可完成

### 5.2 当前工程上的短板

- 没有自动化测试脚本
- 没有单元测试或集成测试用例
- 缺少对真实设备行为的回归验证记录
- 缺少更明确的发布前检查清单
- `app/main/mapping/store.ts` 仍为空，映射存储层尚未单独抽象

## 6. 下一步最值得做的优化方向

当前最有价值的工作，不是继续堆新页面，而是把现有主链路做得更清晰、更稳定、更好用。

### 6.1 第一优先级：补稳定性基础

建议优先补：

- 共享模型测试
- 规则归一化测试
- move 解析器测试
- 紧急停止与执行队列行为测试

目标：

- 降低后续改动风险
- 提升对协议解析与动作执行的信心

### 6.2 第二优先级：优化映射编辑体验

建议把“字段编辑”继续往“动作卡片编辑”推进：

- 让 move 标签更醒目
- 让当前动作结果一句话可读
- 对 `hold` 的后果给出更明确提示
- 降低首次配置时的理解成本

目标：

- 提升可用性
- 减少工程化表单感

### 6.3 第三优先级：继续收口首页

建议让首页进一步专注于：

- 当前是否已连接
- 当前系统是否运行
- 当前激活的是哪套方案
- 连接 / 断开
- 启动 / 暂停
- 紧急停止

把更多配置细节留在各自页面。

目标：

- 让首页真正承担“状态判断 + 快速控制”职责

### 6.4 第四优先级：把诊断页做得更面向普通用户

建议在现有基础上继续增强：

- 当前连接是否健康
- 最近一次失败发生在哪一步
- 推荐下一步操作
- 原始日志作为折叠的详细层

目标：

- 减少用户必须阅读技术日志的压力
- 让诊断页既能服务普通用户，也能保留开发信息

### 6.5 第五优先级：扩展动作与场景能力

在主链路更稳之后，再考虑：

- 组合键
- 多步宏
- 更多鼠标行为
- 双转或更多 move 语义
- 按场景分发的 Profile 模板

目标：

- 提升项目可玩性与实用性
- 把 RubiKey 从“有趣控制器”推进到“更灵活的输入映射工具”

### 6.6 第六优先级：补验证与发布资料

建议逐步补齐：

- 已验证 GAN 型号清单
- 已知兼容问题
- 打包验证记录
- 最小发布清单

目标：

- 让项目从“开发者自己知道能跑”进入“别人也知道当前能用到什么程度”

---

## 7. 当前阶段建议

如果只选一条最现实的推进路径，建议按下面顺序继续：

1. 先补测试，稳住底层
2. 再优化映射编辑与首页信息层级
3. 然后增强诊断体验
4. 最后再扩动作模型和兼容范围

这是当前性价比最高的推进方式，因为它优先巩固已经跑通的主链路，而不是过早扩散功能面。
