# RubiKey 当前产品状态与完成度评估

本文基于 `2026-03-23` 仓库内实际代码重新对齐，只保留当前已经落地的能力、明确存在的缺口，以及下一步最值得推进的工作。

## 1. 当前处于什么阶段

RubiKey 当前更适合定义为：

- 可运行的 Windows 桌面原型
- 多品牌接入第一阶段已落地
- 核心映射链路已经打通
- 仍未进入稳定交付阶段

当前主链路已经成立：

`智能魔方 -> 蓝牙连接 -> 协议解析 -> move 归一化 -> Profile 查找 -> 主进程宏执行 -> Windows 输入`

已确认接入范围：

- GAN 智能魔方 `v2 / v3 / v4`
- Moyu32 / 魔域新协议
- 部分 Moyu32 / V10 AI 风格设备的陀螺仪鼠标控制

## 2. 当前已经落地的能力

### 2.1 连接与设备接入

- 统一接入层 `cube/core` 已落地，GAN 与 Moyu32 共用同一连接工厂
- 支持通过系统蓝牙选择器连接设备
- 支持广告包自动读取 MAC，失败时手动输入兜底
- GAN 已实现 `v2 / v3 / v4` 协议分流
- Moyu32 已实现初始化请求、move 解析与 gyro 包解析

### 2.2 映射与执行

- 支持 12 个基础 move
  - `U / U'`
  - `R / R'`
  - `F / F'`
  - `D / D'`
  - `L / L'`
  - `B / B'`
- 支持多 Profile 持久化
- 支持每个 move 绑定多步宏
- 单步支持顺序执行或同时触发
- 动作类型支持键盘 / 鼠标
- 键盘目标支持 `A-Z`、`0-9`、方向键、`Shift / Ctrl / Alt / Space / Enter / Tab / Esc / Backspace`
- 鼠标目标支持左键 / 右键
- 行为支持 `tap` / `hold`
- 支持应用内录制键盘宏
- 紧急停止会中断执行并释放已按下输入

### 2.3 主进程与桌面能力

- 主窗口可运行，关闭后隐藏到托盘
- 托盘菜单支持显示/隐藏、启停系统、紧急停止、退出
- 已注册全局快捷键
  - `Ctrl/Cmd + Shift + F11`：切换运行状态
  - `Ctrl/Cmd + Shift + F12`：紧急停止
- preload / IPC 边界已接好，Profile、运行态和陀螺仪控制均可从渲染层调用

### 2.4 前端能力

当前前端已经不是单纯的临时连接面板，而是一个单窗口控制台。已落地页面与能力包括：

- 首页
  - 连接状态、设备名、协议、MAC
  - 运行启停、急停、手动 MAC 输入
  - 陀螺仪开关、预览与参数配置
- 方案映射
  - 多方案切换、新增、删除
  - 12 个 move 的规则编辑
  - 多步宏编辑、顺序/同时模式切换
  - 应用内键盘宏录制
- 最近转动
- 执行回响
- 连接诊断
- 关于

需要注意的一点是：虽然 UI 形态已经成型，但主组件名仍然沿用早期命名 `CubeDebugPage`，这说明前端已经超出“调试页”阶段，但代码命名还没有完全收口。

## 3. 当前未完成或仍然偏弱的部分

### 3.1 稳定性与连接体验

- 被动断开时，驱动层会记录日志，但渲染层没有统一的“已断开”状态回写
- `select-bluetooth-device` 逻辑仍优先匹配 GAN 风格名称，多品牌体验未完全收口
- MAC 自动获取仍依赖 advertisement，失败时还需要手动输入兜底
- 连接失败、断开恢复、重连路径缺少系统性人工验证记录

### 3.2 工程完备度

- 仓库内没有自动化测试
- 没有 parser、宏执行队列、急停、规则归一化的回归用例
- 没有兼容矩阵、已知问题列表、发布检查清单
- `app/main/mapping/store.ts` 仍为空，映射持久化尚未拆成独立抽象

### 3.3 产品可用性

- 映射编辑器仍偏工程表单，普通用户学习成本较高
- 连接诊断已经能看日志，但“问题 -> 原因 -> 建议动作”还不够完整
- 陀螺仪默认方向、阈值和手感缺少真机矩阵支撑

## 4. 完成度判断

结合当前代码，较保守的完成度判断为：

- 核心主链路：`80% - 88%`
- 前端产品形态：`70% - 78%`
- 工程完备度：`55% - 65%`
- 交付准备度：`50% - 60%`

这样判断的原因是：

- 主链路已经闭环，不再是概念验证
- 多品牌接入和陀螺仪第一阶段都已经实际落地
- 但稳定性验证、异常恢复、自动化测试和发布资料明显不足

## 5. 目前最值得做的事

如果目标是把项目从“可运行原型”推进到“可稳定交付”，建议按这个顺序收口：

1. 补连接状态一致性
   - 被动断开后的 UI 状态同步
   - 多品牌蓝牙选择体验统一
   - 连接失败和恢复提示收口
2. 补底层回归测试
   - `shared/profiles`
   - GAN / Moyu32 parser
   - 宏执行队列与急停
   - gyro 计算逻辑
3. 补兼容与交付资料
   - 已验证设备清单
   - 陀螺仪推荐参数
   - 已知问题与回避方式
   - 发布前检查清单
4. 再继续打磨前端表达和动作模型

## 6. 本轮对齐结论

这轮核对后的结论很明确：

- 项目真实状态已经高于“单一 GAN Demo”
- 但还没有到“能放心交付”的阶段
- 当前最该补的不是再接一个新协议，而是把现有连接一致性、验证深度和交付资料补齐
