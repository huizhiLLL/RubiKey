# Moyu32 扩展现状与收口项

文件名沿用历史命名，但截至 `2026-03-23`，这里记录的是 Moyu32 第一阶段的实际落地情况，而不是待写方案。

## 1. 当前结论

Moyu32 第一阶段的代码闭环已经成立：

- 已进入统一接入层
- 已接入现有 move -> Profile -> 宏执行链路
- 已具备 V10 AI 风格陀螺仪鼠标的第一阶段支持

当前主要缺的不是继续搭框架，而是验证、状态同步和交付资料。

## 2. 已落地范围

### 2.1 架构接入

- `cube/core` 统一工厂已经落地
- Moyu32 驱动位于 `cube/moyu32`
- 渲染层通过同一连接工厂消费 move / gyro / debug 事件

### 2.2 协议范围

当前已覆盖：

- 设备名前缀 `WCU_MY3*`
- service / read / write characteristic
- 初始化请求 `161 / 163 / 164`
- move 消息 `165`
- gyro 数据 `171 / 0xAB`
- gyro 控制 `172 / 0xAC`

### 2.3 上层可用能力

- Moyu32 转动可进入最近转动日志
- 对应 move 可以触发 Profile 匹配与宏执行
- 连接诊断中可看到品牌、协议、设备名、MAC
- 对支持陀螺仪的设备，可以请求启用 gyro
- gyro 数据已同时送入渲染层预览与主进程鼠标控制器

## 3. 当前边界

第一阶段当前明确不包含：

- facelet 完整状态同步
- 电量与硬件信息的完整 UI 展示
- 历史 move 补偿与丢包恢复
- 多设备并行连接
- 双转语义上抛

这些都不该再混入“第一阶段是否完成”的判断里。

## 4. 当前已知短板

### 4.1 连接与状态

- 被动断开时，驱动层会打日志，但 UI 状态没有统一回写
- MAC 自动获取仍依赖 advertisement，失败时需要手动输入
- 主进程蓝牙设备选择仍带有 GAN 名称优先逻辑，多品牌体验还没完全对齐

### 4.2 真机验证

- 陀螺仪默认轴向与实际握持直觉仍需真机确认
- 不同阈值和步长在不同设备上的手感还缺验证矩阵
- 急停、暂停、断开场景下的 gyro 边界行为还缺系统性回归

## 5. 当前完成度判断

较保守的判断为：

- 架构接入完成度：`90%+`
- 协议接入完成度：`80% - 90%`
- 真机验证完成度：`40% - 55%`
- 交付准备度：`45% - 55%`

这意味着 Moyu32 不是“还没接进去”，而是“已经接进去，但还没收口”。

## 6. 后续收口建议

优先级建议如下：

1. 补被动断开后的状态同步
2. 统一多品牌蓝牙选择体验
3. 补基础真机验证记录
4. 整理已验证型号、参数建议和已知问题

## 7. 补充说明：陀螺仪双模式

当前陀螺仪鼠标已经区分两种模式：

- `desktop`
  - 连续位移
  - 更偏桌面指针控制
- `game`
  - 使用更慢的脉冲式位移
  - 更适合第一人称视角类游戏

这个模式切分已经在共享配置、渲染层设置和主进程控制器中实际落地。
